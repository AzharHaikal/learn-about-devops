DevOps Fundamentals

What is DevOps?

DevOps is a set of practices that combines software development (Dev) and IT operations (Ops). It aims to shorten the systems development life cycle and provide continuous delivery with high software quality. DevOps is about cultural change, automation, collaboration, and integration across all phases of the software delivery pipeline.

DevOps Culture: CAMs

  • Collaboration: Breaking down silos between development, operations, and other teams.
  • Automation: Automating repetitive tasks across the entire SDLC, from testing to deployment.
  • Measurement: Tracking key metrics (e.g., deployment frequency, lead time, mean time to recovery) to identify bottlenecks and improve processes.
  • Sharing: Sharing knowledge, tools, and best practices across teams to foster continuous learning and improvement.

SDLC vs DevOps Lifecycle

Traditional SDLC (Software Development Life Cycle) often involves distinct, sequential phases with handoffs between teams, leading to slower delivery and potential communication gaps. The DevOps lifecycle, in contrast, is a continuous loop emphasizing iteration, feedback, and automation across all stages.

Feature Traditional SDLC DevOps Lifecycle
Phases Sequential (Plan, Design, Implement, Test, Deploy, Maintain) Continuous (Plan, Code, Build, Test, Release, Deploy, Operate, Monitor)
Team Interaction Siloed, Handoffs Collaborative, Integrated
Delivery Speed Slower, Infrequent Faster, Frequent
Feedback Loop Delayed Continuous, Early
Automation Limited Extensive

Continuous Integration & Delivery (CI/CD)

Concepts: CI, CD (Delivery & Deployment)

  • Continuous Integration (CI): Developers frequently merge their code changes into a central repository, after which automated builds and tests are run.
  • Continuous Delivery (CD): Ensures that code changes are automatically built, tested, and prepared for release to production. It ensures that you can release new changes to your customers rapidly and sustainably.
  • Continuous Deployment (CD): An extension of Continuous Delivery, where every change that passes all stages of the production pipeline is released to production automatically, without human intervention.

Tools

  • Jenkins: Open-source automation server for building, testing, and deploying software.
  • GitLab CI/CD: Integrated CI/CD solution within GitLab, using YAML for pipeline definitions.
  • GitHub Actions: Event-driven automation platform directly integrated with GitHub repositories.
  • CircleCI: Cloud-based CI/CD platform known for its speed and ease of use.
  • ArgoCD: Declarative GitOps continuous delivery tool for Kubernetes.

Example CI/CD YAML Pipeline (GitHub Actions)


name: CI/CD Pipeline

on:
  push:
    branches:
      - main
  pull_request:
    branches:
      - main

jobs:
  build-and-test:
    runs-on: ubuntu-latest
    steps:
    - name: Checkout code
      uses: actions/checkout@v3

    - name: Set up Node.js
      uses: actions/setup-node@v3
      with:
        node-version: '18'

    - name: Install dependencies
      run: npm install

    - name: Run tests
      run: npm test

    - name: Build application
      run: npm run build

    - name: Upload artifact
      uses: actions/upload-artifact@v3
      with:
        name: build-output
        path: build/ # Assuming 'build' directory contains compiled app

  deploy:
    needs: build-and-test
    runs-on: ubuntu-latest
    environment: production # Define an environment for deployment
    steps:
    - name: Download artifact
      uses: actions/download-artifact@v3
      with:
        name: build-output
        path: ./app-dist

    - name: Deploy to server (example using rsync)
      run: |
        echo "Deploying application to production server..."
        # Replace with actual deployment commands (e.g., rsync, kubectl apply, etc.)
        # rsync -avz ./app-dist/ user@your-server:/var/www/html/
        echo "Deployment complete!"

Canary vs Blue-Green Deployment Strategies

  • Canary Deployment: A new version of the application (the "canary") is rolled out to a small subset of users. If no issues are detected, it's gradually rolled out to the rest of the users. This minimizes risk and allows for real-world testing.
  • Blue-Green Deployment: Two identical production environments exist: "Blue" (current live version) and "Green" (new version). Traffic is switched from Blue to Green once the Green environment is fully tested and ready. This provides zero-downtime deployment and easy rollback by switching traffic back to Blue.

Monolith vs Microservices

Architecture Comparison

Feature Monolithic Architecture Microservices Architecture
Structure Single, tightly coupled unit Collection of small, independent services
Development Slower for large teams, single codebase Faster for large teams, independent development
Deployment Deploy entire application for any change Deploy individual services independently
Scalability Scale entire application, often inefficient Scale individual services based on demand
Technology Stack Single technology stack Polyglot persistence, diverse technologies
Fault Isolation Failure in one part can affect entire app Failure in one service typically isolated

When to use Microservices

  • For large, complex applications that need to evolve rapidly.
  • When different parts of the application have varying scaling requirements.
  • When multiple independent teams need to work on different parts of the system concurrently.
  • To leverage diverse technology stacks for different functionalities.
  • When fault isolation is critical to overall system resilience.

Service Discovery and API Gateway

  • Service Discovery: In a microservices architecture, services need to find and communicate with each other. Service discovery mechanisms (e.g., Eureka, Consul, Kubernetes DNS) help services locate each other dynamically.
  • API Gateway: A single entry point for all client requests to a microservices system. It handles request routing, composition, and protocol translation. Examples include Kong and Istio (which is a service mesh, offering API Gateway capabilities among others).

Containerization

Docker Basics

  • Image: A lightweight, standalone, executable package that includes everything needed to run a piece of software, including the code, a runtime, system tools, system libraries, and settings.
  • Container: A runnable instance of an image. Containers are isolated from each other and the host system, but can communicate through defined channels.
  • Volume: A mechanism for persisting data generated by and used by Docker containers. Volumes are preferred over bind mounts for most use cases because they are fully managed by Docker.
  • Network: Docker networking allows containers to communicate with each other and with the outside world. Docker provides several networking drivers (bridge, host, overlay, etc.).

Best Practices for Writing Dockerfiles

  • Use a small base image (e.g., Alpine Linux).
  • Use multi-stage builds to separate build-time dependencies from runtime dependencies.
  • Leverage Docker's build cache effectively.
  • Minimize the number of layers.
  • Copy only necessary files.
  • Use `.dockerignore` to exclude unnecessary files.
  • Run as a non-root user.
  • Define explicit `CMD` or `ENTRYPOINT`.

Docker Compose Usage

Docker Compose is a tool for defining and running multi-container Docker applications. You use a YAML file to configure your application's services, networks, and volumes, then use a single command to start everything up.


version: '3.8'

  web:
    build: .
    ports:
      - "8000:8000"
    volumes:
      - .:/code
    depends_on:
      - db
  db:
    image: postgres:13
    environment:
      POSTGRES_DB: mydb
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password

Container Registry Examples

  • DockerHub: The world's largest library and community for container images.
  • AWS ECR (Elastic Container Registry): A fully managed Docker container registry that makes it easy for developers to store, manage, and deploy Docker container images.
  • Google Container Registry (GCR)/Artifact Registry: Google Cloud's service for storing and managing container images.

Orchestration

Kubernetes Core Concepts

  • Pods: The smallest deployable units of computing that can be created and managed in Kubernetes. A Pod typically contains one or more containers.
  • Deployments: Manages a replicated set of Pods. It provides declarative updates to Pods and ReplicaSets.
  • Services: An abstract way to expose an application running on a set of Pods as a network service.
  • Ingress: An API object that manages external access to the services in a cluster, typically HTTP.
  • ConfigMap: Used to store non-confidential data in key-value pairs.
  • Secrets: Used to store sensitive information, such as passwords, OAuth tokens, and ssh keys.

Helm (Kubernetes Package Manager)

Helm helps you manage Kubernetes applications. Helm Charts are packages of pre-configured Kubernetes resources. Helm simplifies the deployment and management of complex applications on Kubernetes.

Minikube vs EKS/GKE/AKS

  • Minikube: A tool that runs a single-node Kubernetes cluster on your local machine, ideal for development and testing.
  • EKS (AWS Elastic Kubernetes Service), GKE (Google Kubernetes Engine), AKS (Azure Kubernetes Service): Managed Kubernetes services offered by major cloud providers. They handle the control plane management, making it easier to run production-grade clusters.

Kubernetes Observability

  • Metrics: Quantitative data about the behavior of a system (e.g., CPU usage, memory consumption, request latency). Prometheus is a common tool for collecting and storing metrics.
  • Logs: Timestamped records of events that happen within a system. Tools like Fluentd, Loki, and the ELK stack are used for log aggregation and analysis.
  • Tracing: Following the path of a single request as it propagates through multiple services in a distributed system. Tools like Jaeger and Zipkin are used for distributed tracing.

Infrastructure as Code (IaC)

Tools

  • Terraform: An open-source IaC tool by HashiCorp that allows you to define and provision infrastructure using a declarative configuration language (HCL). Supports multiple cloud providers.
  • AWS CloudFormation: AWS's native IaC service for modeling and provisioning AWS resources.
  • Pulumi: An open-source IaC tool that allows you to define infrastructure using familiar programming languages (Python, JavaScript, TypeScript, Go, C#).

Monitoring & Logging

Monitoring Tools

  • Grafana: An open-source platform for monitoring and observability. It allows you to query, visualize, alert on, and understand your metrics no matter where they are stored.
  • Prometheus: An open-source monitoring system with a dimensional data model, flexible query language (PromQL), and powerful alerting capabilities. Often used as a data source for Grafana.

Logging

  • Loki: A horizontally-scalable, highly-available, multi-tenant log aggregation system inspired by Prometheus. It focuses on storing logs as streams and querying them using labels.
  • Fluentd: An open-source data collector for unified logging layer. It allows you to collect, parse, transform, and output logs to various destinations.
  • Promtail: An agent that ships the contents of local logs to a private Loki instance or Grafana Cloud.

ELK Stack

The ELK Stack is a collection of three open-source products: Elasticsearch, Logstash, and Kibana. It's commonly used for centralized logging, allowing you to search, analyze, and visualize logs from various sources.

  • Elasticsearch: A distributed, RESTful search and analytics engine capable of solving a growing number of use cases.
  • Logstash: A server-side data processing pipeline that ingests data from multiple sources simultaneously, transforms it, and then sends it to a "stash" like Elasticsearch.
  • Kibana: A free and open user interface that lets you visualize your Elasticsearch data and navigate the Elastic Stack.

Alerting

  • Alertmanager: Handles alerts sent by client applications such as the Prometheus server. It takes care of deduplicating, grouping, and routing them to the correct receiver integration.
  • Opsgenie: A modern incident management platform that ensures critical alerts are never missed and notifies the right people through multiple channels.
  • PagerDuty: An incident management platform that provides on-call scheduling, alerting, and escalation to help teams respond to critical incidents faster.

Security & Compliance (DevSecOps)

DevSecOps Principles

DevSecOps integrates security practices into every phase of the DevOps pipeline. The core principle is "shift left," meaning security is considered from the very beginning of the development process, rather than being an afterthought.

  • Automate security testing (SAST, DAST, SCA).
  • Treat security as code.
  • Implement continuous monitoring for security vulnerabilities.
  • Foster a culture of shared security responsibility.

Secrets Management

  • Vault (HashiCorp): A tool for securely accessing secrets. A secret is anything that you want to tightly control access to, such as API keys, passwords, certificates.
  • AWS Secrets Manager: A service that helps you protect access to your applications, services, and IT resources by enabling you to easily rotate, manage, and retrieve database credentials, API keys, and other secrets throughout their lifecycle.

Container Image Scanning

  • Trivy: A comprehensive and easy-to-use vulnerability scanner for containers and other artifacts.
  • Anchore: Provides deep inspection, compliance, and vulnerability scanning for container images.

Access Control

  • IAM (Identity and Access Management): A web service that helps you securely control access to AWS resources.
  • Kubernetes RBAC (Role-Based Access Control): A method of regulating access to computer or network resources based on the roles of individual users within an enterprise. In Kubernetes, it allows you to define who can do what within the cluster.

Cloud Providers

IAM & Roles

Identity and Access Management (IAM) is fundamental to cloud security. It allows you to manage access to cloud resources securely. Roles are a secure way to grant permissions to entities that you trust (e.g., EC2 instances, Lambda functions, or other AWS services) without having to manage credentials.

Common Services (AWS Examples)

  • EC2 (Elastic Compute Cloud): Virtual servers in the cloud.
  • S3 (Simple Storage Service): Object storage for data archiving, backup, and big data analytics.
  • RDS (Relational Database Service): Managed relational databases (MySQL, PostgreSQL, etc.).
  • Lambda: Serverless compute service that runs code in response to events.

ECS vs EKS

  • ECS (Elastic Container Service): AWS's proprietary container orchestration service. Simpler to set up for Docker containers if you are already in the AWS ecosystem.
  • EKS (Elastic Kubernetes Service): AWS's managed Kubernetes service. Offers the power and flexibility of Kubernetes, which is cloud-agnostic and widely adopted.

Cloud Networking (AWS Examples)

  • VPC (Virtual Private Cloud): A virtual network dedicated to your AWS account.
  • Subnets: Logical subdivisions of a VPC, allowing you to segment your network.
  • Security Groups: Act as virtual firewalls for your instances to control inbound and outbound traffic.

Cost Optimization & Tagging

Efficient cloud usage involves managing costs. Tagging resources (e.g., by project, owner, environment) is crucial for cost allocation, reporting, and automation. Other strategies include right-sizing instances, utilizing reserved instances/savings plans, and monitoring spending with cloud cost management tools.

Version Control (Git)

Git Basics

  • Commit: A snapshot of your repository at a specific point in time.
  • Branch: A lightweight movable pointer to a commit. Used to isolate development work.
  • Merge: Combining changes from one branch into another.
  • Pull Request (PR)/Merge Request (MR): A mechanism to propose changes to a repository and discuss them before merging.

Git Workflows

  • Gitflow: A branching model that defines strict rules for branching and merging, often used for projects with scheduled releases. Involves `main`, `develop`, `feature`, `release`, and `hotfix` branches.
  • Trunk-Based Development: Developers merge small, frequent updates to a single main branch (the "trunk"). This promotes continuous integration and faster feedback.

Pull Request Best Practices

  • Keep PRs small and focused on a single logical change.
  • Write clear and concise descriptions, explaining "what" and "why."
  • Include relevant context (e.g., linked issues, screenshots).
  • Request reviews from appropriate team members.
  • Respond to feedback promptly and constructively.

Automation & Scripting

Shell Scripting

Shell scripting (e.g., Bash) is fundamental for automating repetitive tasks on Linux/Unix systems. It's used for file operations, system administration, executing commands, and orchestrating simple workflows.


#!/bin/bash
# Simple script to backup a directory
SOURCE_DIR="/var/www/html"
DEST_DIR="/mnt/backups"
TIMESTAMP=$(date +"%Y%m%d%H%M%S")
BACKUP_FILE="${DEST_DIR}/web_backup_${TIMESTAMP}.tar.gz"

echo "Starting backup of ${SOURCE_DIR}..."
tar -czf "$BACKUP_FILE" "$SOURCE_DIR"
echo "Backup complete: ${BACKUP_FILE}"

Python Scripting for Automation

Python is widely used in DevOps for its versatility, rich ecosystem of libraries, and readability. It's excellent for automating complex tasks, interacting with APIs (cloud providers, monitoring systems), data processing, and building custom tools.


# simple_api_call.py
import requests

def get_github_user_info(username):
    url = f"https://api.github.com/users/{username}"
    response = requests.get(url)
    if response.status_code == 200:
        user_data = response.json()
        print(f"User: {user_data['login']}")
        print(f"Name: {user_data.get('name', 'N/A')}")
        print(f"Public Repos: {user_data['public_repos']}")
    else:
        print(f"Error fetching user info: {response.status_code}")

if __name__ == "__main__":
    get_github_user_info("octocat")

Release Management & Deployment Strategies

Deployment Strategies

  • Rolling Update: Gradually replaces instances of the old version of an application with instances of the new version. This minimizes downtime and risk.
  • Recreate: Terminates all old instances before creating new ones. Simplest but causes downtime.
  • Blue-Green: (Reiterated from CI/CD) Maintain two identical environments; switch traffic to the new one. Zero downtime, easy rollback.
  • Canary: (Reiterated from CI/CD) Roll out to a small subset of users first. Gradual rollout, risk mitigation.

Feature Toggles (Feature Flags)

Feature toggles allow you to turn features on or off without deploying new code. They decouple deployment from release, enabling A/B testing, gradual rollouts, and quick rollbacks of problematic features.

  • Benefits: Decouple deployment from release, A/B testing, gradual rollouts, instant rollback of features, dark launching.
  • Implementation: Typically involves a configuration service or database that stores the state of each toggle, and application code that checks this state before executing a feature.

Real-World Case Studies

CI/CD for a Spring Boot Application

A typical CI/CD pipeline for a Spring Boot application might involve:

  • Commit: Developer pushes code to Git (e.g., GitHub).
  • CI Trigger: GitHub Actions (or Jenkins/GitLab CI) detects the push.
  • Build: Maven/Gradle builds the JAR/WAR file.
  • Test: Unit, integration, and perhaps static code analysis (SonarQube) run.
  • Containerize: Dockerfile builds a Docker image of the Spring Boot app.
  • Push to Registry: Docker image is pushed to DockerHub/ECR.
  • CD Trigger: Successful image push triggers deployment.
  • Deploy: Kubernetes Deployment updates the application Pods, potentially using a rolling update strategy.
  • Monitor: Prometheus/Grafana monitor application health and performance.

Migrating from Monolith to Microservices

This complex process often involves:

  • Identify Bounded Contexts: Decompose the monolith into logical, independent services based on business capabilities.
  • Strangler Fig Pattern: Gradually replace parts of the monolith with new microservices, routing traffic to the new services while the old functionality is "strangled."
  • Data Migration/Synchronization: Carefully plan how data will be handled across services (e.g., shared database vs. separate databases, event-driven consistency).
  • Implement API Gateway & Service Discovery: Essential for managing communication in the new distributed system.
  • Automate Deployment & Monitoring: Critical for managing the increased complexity of many services.

Production-Grade Observability Practices

Ensuring your production systems are observable is key to quickly identifying and resolving issues:

  • Standardized Logging: Use structured logging (JSON) and centralize logs with tools like ELK or Loki.
  • Comprehensive Metrics: Collect metrics from applications (e.g., JVM metrics, request latency), infrastructure (CPU, memory), and network. Use Prometheus exporters.
  • Distributed Tracing: Implement tracing (e.g., OpenTelemetry, Jaeger) to visualize request flows across microservices.
  • Effective Alerting: Set up meaningful alerts based on critical metrics and logs, integrated with on-call systems (PagerDuty).
  • Dashboards: Create intuitive Grafana dashboards for different teams (developers, operations) to visualize system health.
  • Blameless Postmortems: Conduct post-incident reviews to learn from failures and improve systems and processes.
